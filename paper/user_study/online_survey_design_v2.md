# EnvMeta 用户体验问卷设计 v2（双轨紧凑版）

> **目标受众**：双轨设计 —— ① 非生信背景用户（其他专业研究生 / 本科生）测基础交互；② 有生物或生信基础用户多做 1 页深度任务
> **形式**：腾讯问卷，电子化纯线上，无需主持人介入
> **总耗时**：**12-15 min（非生信轨道）/ 22-25 min（深度轨道）**，所有人 ≤ 30 min
> **题量**：**33 题**（v1 是 87 题，砍掉 62%；其中 4 题为可跳过的深度任务）
> **实际导入文件**：[tencent_survey_import_v2.txt](tencent_survey_import_v2.txt) —— 这是腾讯问卷后台直接 import 用的纯文本，本 .md 是设计规约
> **设计版本**：v2，2026-05-07 制定 + 双轨调整，源于 v1 反馈"太长难以答完" + 用户群体异质（非生信背景多）
>
> **v1 vs v2**：v1 [online_survey_design.md](online_survey_design.md) 保留作为
> 历史快照（n=2 反馈见 [v1_archived_issues.md](v1_archived_issues.md)）。
> v2 是实际重发内测群用的版本。
>
> **单轨 → 双轨调整**：原 v2 单轨设计（24 题 / 12-15 min）面向"环境微生物方向研究生"，但实测内测群同学背景非常异质（含化学、材料、计算机方向同学）。
> 强行让所有人答 PCoA 解读 / 假说评分等深度任务会被中途放弃。
> 双轨改造：所有人共同做基础交互任务（任务 1-4），仅生信基础用户额外做深度任务（任务 5-6 假说 YAML / PCoA 深度解读）。

## 设计原则（v2 调整后）

| 原则 | v1 | v2 调整 |
|---|---|---|
| 问卷长度 | 87 题 / 30-40 min | **24 题 / 12-15 min** |
| 强制截图 | 5 任务都要截图 | **只 1 张最终循环图** |
| 知识测验 | 上游工具 mapping 选择题 3 道 | **全砍**（不是可用性测试该测的内容） |
| 任务自评 | 每任务 5-8 道 Likert + 多选 | **每任务 1 完成度 + 1 难度** |
| 开放题 | 12 道 | **3 道**（最受挫 / 最惊喜 / 想说的） |
| Likert 矩阵 | 多组 1-5 评分 | **保留 SUS 10 题 + NPS 1 题，其他全砍** |
| Multi-select 多选 | 多组（最多 11 选项） | **全砍** |
| 模块覆盖 | 8 任务 | **5 任务**（去掉数据准备 + 假说评分作为可选） |

## 核心思想

**问卷的目的是测可用性（usability），不是 KPI 全维度评估**。所以 v2 只保留：
- 国际通用的 **SUS** 量表（10 题，跨工具对照用）
- **NPS** 净推荐值（1 题，论文核心采纳指标）
- **任务完成率** + **每任务难度**（5 任务 × 2 题 = 10 题）
- 3 道开放题捕捉意外洞察

其他细颗粒度反馈（具体功能 priority / 想加的图 / UI tooltip 评分）→ 改用 GitHub Issues / 邮件反馈持续收集，**不应该塞进一次性问卷**。

### 页 3 — 5 个核心任务（每任务 2 题，共 10 题，5-7 分钟）

**前置说明**：
```
按下面 5 个任务依次操作。每个任务后只问 2 个问题：
  ① 是否完成（是 / 否）
  ② 难度（1=非常容易 / 5=非常难）

🎯 任务**全部失败也算有效反馈**，请如实填。
```

#### 任务 1：加载样例数据

**操作**：首页点「📦 加载砷渣修复示例数据」按钮。

**Q4** 完成情况（单选）
- ✅ 完成
- ❌ 找不到按钮 / 加载失败

**Q5** 难度（1-5）

#### 任务 2：跑 PCoA 分析

**操作**：左侧栏选 「β 多样性 PCoA」→ 点「生成图」→ 看「如何解读」expander。

**Q6** 完成情况
- ✅ 看到 PCoA 图 + 读完解读 expander
- ⚠️ 生成了图但没读 expander
- ❌ 没生成出图

**Q7** 难度（1-5）

#### 任务 3：交互 HTML 循环图

**操作**：左侧栏「循环图」→ 生成 → **导出独立交互 HTML 并打开**。
在 HTML 里**拖拽一个化学物节点 + 点击一个 MAG 节点看高亮效果**。

**Q8** 完成情况
- ✅ 拖拽 + 点击都成功
- ⚠️ 拖拽成功，点击没看到高亮
- ⚠️ 找到导出 HTML 按钮但拖拽 / 点击没尝试
- ❌ 没找到「导出 HTML」入口

**Q9** 难度（1-5）

#### 任务 4：修改一个图的样式 + 导出 PDF

**操作**：任选一张图 → 改字体或配色 → 导出中心下载 PDF。

**Q10** 完成情况
- ✅ 改了样式 + 拿到 PDF
- ⚠️ 拿到 PDF 但没改样式
- ❌ 找不到导出中心 / 改样式没生效

**Q11** 难度（1-5）

#### 任务 5：（可选）假说 YAML 评分

> ⚠️ **可跳过本任务**直接到下一页，不影响整体反馈。

**操作**：上传一份 YAML（`paper/hypotheses/arsenic_steel_slag.yaml` 已示例）→ 点评分。

**Q12** 完成情况
- ✅ 拿到 overall label + 9 档解读
- ⚠️ 上传成功但看不懂结果
- ❌ 上传失败 / 跳过此任务

**Q13** 难度（1-5，跳过填 0）

### 页 5 — SUS 量表（10 题，强制必填，3 分钟）

> 国际标准可用性量表 (System Usability Scale)。
> 1=非常不同意 / 5=非常同意

| # | 题目 |
|---|---|
| Q15 | 我觉得我会经常使用这个工具 |
| Q16 | 我觉得这个工具不必要地复杂 |
| Q17 | 我觉得这个工具很容易使用 |
| Q18 | 我觉得需要技术人员的帮助才能使用这个工具 |
| Q19 | 我觉得工具的各项功能整合得很好 |
| Q20 | 我觉得工具里有太多不一致的地方 |
| Q21 | 我觉得大多数人都能很快学会使用这个工具 |
| Q22 | 我觉得这个工具用起来很麻烦 |
| Q23 | 使用这个工具时我很有信心 |
| Q24 | 我需要先学很多东西才能开始使用这个工具 |

> **后台计算**（不展示给用户）：
> - 奇数题（Q15/17/19/21/23）得分 = 原始分 - 1
> - 偶数题（Q16/18/20/22/24）得分 = 5 - 原始分
> - SUS = (10 题总和) × 2.5；满分 100，> 68 = Good，> 80 = Excellent

## 实施步骤（双轨版）

1. **腾讯问卷后台**：新建问卷 → "导入题库 / 文本批量导入" → 粘贴 [tencent_survey_import_v2.txt](tencent_survey_import_v2.txt) 全文
2. **配置跳题逻辑**（双轨核心）：
   - 找到第 4 题"你的生物或生信基础"
   - 选项 1（"完全无生物背景"）和选项 2（"有生物背景但不熟生信"）→ 设置跳过到"页 7 截图上传"，**绕过页 6 深度任务**
   - 选项 3（"学过微生物或基础生信"）和选项 4（"熟练生信"）→ 正常进入页 6
3. **检查页面顺序**：1 背景 → 2-5 任务 1-4（所有人）→ 6 深度任务（仅 C/D 类用户）→ 7 截图 → 8 SUS → 9 NPS+开放
4. **必填检查**：除 SUS 10 题 + 任务 1-4 完成度 + Q4 背景类型 + 截图，其余设为选填
5. **预览跑一遍**：自己以"非生信用户"身份和"生信用户"身份各跑一遍，验证跳题逻辑生效
6. **生成新链接**：发到内测群时**强调"双轨：12-15 min 或 22-25 min，自动判断"**

## 投稿引用模板（论文 Methods 里这么写）

```
We conducted an online usability survey using Tencent Survey
(https://wj.qq.com), targeting graduate students in environmental
microbiology. Participants completed five core tasks (loading sample
data, beta-diversity PCoA analysis, interactive HTML cycle figure,
plot styling and PDF export, and an optional hypothesis YAML scoring),
followed by the standard 10-item System Usability Scale (SUS; Brooke,
1996), a Net Promoter Score item (Reichheld, 2003), and three
open-ended questions. The survey was designed to take 12-15 minutes
and was iteratively shortened from a longer pilot (87 items, 30-40
min) after pilot respondents (n=2) reported fatigue.
```
